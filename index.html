绕过的一些语句功能特性

  • selectCHAR

SQL查询语句slect后面可以接一些特殊字符，这些字符与select相结合可以达到绕Waf的目的，除了select语句之外union\from等关键字前后也可以连接一些特殊字符，这些关键字前后就可以作为fuzz的点。

等价替换

  • 函数替换

    1. 截取字符串是 SQL注入中的常用功能，通常使用mid(string,1,1)

      sekect mid(user(),1,1);

    2. usbstr(user() from 1 for 1);
    3. SELECT replace(LPAD(user(),2,1),LPAD(user(),2-1,1),””);
    4. SELECT LPAD(REVERSE(TRIM(lpad(user(),1,SPACE(1)))),1,SPACE(1));
    5. ascii(c)、ord(c)<=>conv(hex(c),16,10)

      select ord(“t”);

  • 逗号过滤

    逗号在被waf拦截或过滤时，通过不含引号的sql语句代替。

    1. case when 代替if
    2. union select 1,2,3<=>union select*from (select 1)a join (select 2)b join (select 3)c
    3. limit 2,1 <=>limit 1 offset 2

  • 比较表达式代替
    代替【=】
    1.if(abs(strcmp((ascii(mid(user()from(1)for(2)))),114))-1,1,0)
    2.find_in_set()

       > select find_in_set(ord(mid(user() from 1 for 1)), 114);
   > select find_in_set(ord(mid(user() from 1 for 1)), 115);
   
   前者得到的结果为1，后者得到的结果为0，查询114（root）中r的结果。

    3.regexp

       > select ord("r") regexp 114;

目标信息收集

前言

  • 在真实的网络安全环境中，攻击方投入最大的地方往往不是渗透与破坏阶段，而现实的重点都在对目标的信息收集方面；应对现实中真实存在的网络威胁，对目标去做一些提前的分析是可以大致判断出对手或者攻击方的真实实力，从而做出合理的防御措施会更加有益于受害方做出恰当的投入水平。二者在网络攻防领域都是最重要阶段。
  • 本次培训旨在为大家从一个真实的攻击前段流程中看到攻击方会采用何种攻击方式，防守方又会有什么可能的方向去应对这种难题。

    目标

  • 本次的针对多类目标采用多种方式收集信息，主要目标有：
    1. 公民
    2. 网站
    3. 公司体
  • 对应每种类型的目标采取的方式各不相同，以上是相对而言容易剥离开的实体。

    公司主体的信息收集

  • 公司主体的信息收集需要明确的前提是信息收集的颗粒度、搜索的范围边界值问题。比如：对于攻击者而言，重点在整个企业的网络架构、网络硬件设备信息、内网域相关信息、DMZ这种；而对与做画像来说，更倾向于收集IP段，公司域名，DNS，企业地理信息等。在明确了目标后，就可以进行公司的信息收集了。

  • 公民的信息收集

  • 从事正常行业的公民一般不会脱离目前的互联网，对攻击者或渗透而言，公民这个实体要素有：真实姓名、网络昵称、账号、账号名、密码、手机号、身份证号、邮箱、银行卡等。国内和国外目前各自拥有不同的社交圈和收集方式，本次以国内的公民为例。
    1. 黑市
    2. 社工
    3. 网络搜集

威胁分析

前言

  • 威胁分析是一个很大的领域，本次仅从我们现有的雷达平台、流量威胁平台得到的目标去做关于APT威胁的分析，该分析涉及到的领域应为流量分析的前段-也就是不对样本文件做逆向等内核分析，只从该目标所呈现的要素去分析。

    APT

    通过该平台可以得到的威胁分析信息不完全包括使用中不完全包含：
字段 可利用价值
url 高
IP端口 高
payload 特定情况
哈希值 中等

地区 中等
协议信息 高
别的信息在其他平台利用率不高，只针对辅助威胁分析。

分析思路

  • 应当注意的是，如果去仔细研究APT的攻击流程，会发现突破的点大概率在威胁的后半程，也就是攻击利用完针对它的“尸体”去做利用，因为纵观所有的APT攻击，特点就是它的持续性和目标单一性。一次正常的APT威胁，除非受害方愿意公开其内网平台进行白盒审计，不然可以得到的只有其“尸体”。
  • APT在攻击完后留下的“尸体”是我们重点关注的部分，它可以剥离开的要素有CC通信所留下的IP、文件本身、文件的哈希值、木马在编写中使用的代码语法习惯、回连或者水坑过程中使用的域名、钓鱼中使用的公私证书。除此之外，还有就是每次攻击的思路和流程。
  • 判断APT威胁分析的重点就是发现“尸体”的时间，在攻击方做完攻击流程之后，被控制的肉鸡会通过恶意代码进行外联，外联过程会使用各种手段，常见的如隐蔽隧道、矿池、C2通信等，这些方式并非单一利用，比如C2通信方式会被改造成利用FTP或者ICMP协议，利用矿池秘密传输数据等。

SSRF

官方解释

SSRF，Server-Side Request Forgery，服务端请求伪造，是一种由攻击者构造形成由服务器端发起请求的一个漏洞。一般情况下，SSRF 攻击的目标是从外网无法访问的内部系统。

漏洞形成的原因大多是因为服务端提供了从其他服务器应用获取数据的功能且没有对目标地址作过滤和限制。

攻击场景

  • 可以对外网、服务器内网、本地进行端口扫描，获取一些服务的banner信息。
  • 攻击运行在内网或本地的应用程序（例如溢出）
  • 对内网的Web应用进行指纹识别，通过访问默认文件实现
  • 攻击内外网的Web应用，主要是使用GET参数就可以实现的攻击（例如struts2、sqli等）
  • 利用file协议读取本地文件等

SSRF漏洞可能出现的场景

  • 能够对外发起网络请求的地方。
  • 从远程服务器请求资源（upload、import、export rss feed）
  • 数据库内置功能
  • webmail收取其他邮箱的邮件
  • 文件处理、编码处理、属性信息处理（ffmpeg、imagemagic、docx、pdf、XML）

    后端常用函数造成此漏洞

  • file-get-contents()
  • fsockopen()
  • curl_exec()

组织SSRF漏洞利用的场景

  • 服务器开启 OpenSSL 无法进行交互利用
  • 服务端需要鉴权（Cookies & User：Pass）不能完美利用
  • 限制请求的端口为 http常用的端口，比如，80,443,8080,8090。
  • 禁用不需要的协议。仅仅允许http和https请求。可以防止类似于 file:///,gopher://,ftp:// 等引起的问题。
  • 统一错误信息，避免用户可以根据错误信息来判断远端服务器的端口状态
